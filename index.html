Vrbo · Cancellations & Modifications

v1 = system-triggered guarantee revision when Vrbo (EG) pushes a new itinerary. One consolidated email carries the diff + updated receipt; cancel routes through a tokenized link — no auth interstitial. v2 = user-initiated edit from the dashboard (dates · location · upgrade · cancel) + optional +$10 trigger upgrade. Vrbo-scoped; other partners reuse the pattern later.

Updated 2026-04-23 · post-James + post-council  — consolidated update email (diff in its own card + locked receipt copy); cancel = 1 step via tokenized link (no separate auth email); 2 UW outcomes (match · can't match); v2 scoped to 4 dashboard entry points + email deeplink.

v1 — system-triggered revision Primary build

EG sends updated itinerary → WP auto-revises guarantee under locked premium, same order ID → customer gets one consolidated email (diff + updated receipt) → accept silently, or tap the in-email cancel link. The link is a tokenized URL → lands directly on final confirm (token = auth, no interstitial). If the cut-off window has passed, the same URL renders a "cancellation window closed" variant that routes to support. Dashboard updates passively.

EG update → auto-revise → consolidated email → silent accept | cancel link → final confirm or cut-off variant → refund or support routing
